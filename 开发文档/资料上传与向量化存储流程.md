# AstrBot 资料上传与向量化存储流程

## 概述

AstrBot 内置了完整的 RAG（检索增强生成）知识库模块。上传的资料会进行**全量向量化存库**，没有选择性跳过。

---

## 一、上传入口

系统提供三种上传方式：

| 入口 | 路由 | 说明 |
|------|------|------|
| 文件上传 | `POST /kb/document/upload` | multipart/form-data，一次最多 10 个文件 |
| URL 导入 | `POST /kb/document/upload/url` | 通过 Tavily Extract API 抓取网页内容 |
| 预分块导入 | `POST /kb/document/import` | JSON 格式，直接传入已切好的文本块（跳过解析和分块） |

文件上传支持格式：`.md` `.txt` `.markdown` `.docx` `.xlsx` `.xls` `.pdf` `.epub`

关键代码：[knowledge_base.py#L658](AstrBot/astrbot/dashboard/routes/knowledge_base.py#L658)

---

## 二、处理管道（5 个阶段）

完整处理流程位于 [kb_helper.py#L210](AstrBot/astrbot/core/knowledge_base/kb_helper.py#L210) 的 `upload_document()` 方法中。

### 阶段 1：解析（Parsing）

根据文件扩展名选择不同的解析器：

| 文件类型 | 解析器 | 策略 |
|----------|--------|------|
| `.md` `.txt` `.markdown` `.docx` `.xlsx` `.xls` | `MarkitdownParser` | 使用 Microsoft MarkItDown 库转为 Markdown |
| `.pdf` | `PDFParser` | 使用 `pypdf` 逐页提取文本，同时提取嵌入图片为 `MediaItem` |
| `.epub` | `EpubParser` | 委托 MarkitdownParser + 清洗（去头尾/脚注） |
| URL | `URLExtractor` | 使用 Tavily Extract API 获取原始内容 |

解析结果统一为 `ParseResult` 数据类，包含 `text` 和 `media` 两个字段。

关键代码：
- 解析器选择：[util.py](AstrBot/astrbot/core/knowledge_base/parsers/util.py)
- PDF 解析：[pdf_parser.py](AstrBot/astrbot/core/knowledge_base/parsers/pdf_parser.py)
- Markitdown 解析：[markitdown_parser.py](AstrBot/astrbot/core/knowledge_base/parsers/markitdown_parser.py)

### 阶段 2：媒体提取

PDF 中提取的图片保存到本地 `<kb_dir>/medias/<kb_id>/<doc_id>/` 目录，元数据写入 SQLite 的 `kb_media` 表。

### 阶段 3：分块（Chunking）

使用 **RecursiveCharacterChunker**（递归字符分割器），按优先级依次尝试分割符：

1. `\n\n` — 段落
2. `\n` — 换行
3. `。` — 中文句号
4. `，` — 中文逗号
5. `. ` — 英文句号
6. `, ` — 英文逗号
7. ` ` — 空格
8. 字符级回退（滑动窗口 + overlap）

**默认配置：**
- `chunk_size = 512` 字符
- `chunk_overlap = 50` 字符
- 每个知识库可独立配置

关键代码：[recursive.py](AstrBot/astrbot/core/knowledge_base/chunking/recursive.py)

### 阶段 4：向量化（Embedding）

**每个 chunk 都会被向量化。** 支持多种嵌入提供商：

| 提供商 | 默认模型 | 向量维度 |
|--------|----------|----------|
| OpenAI | `text-embedding-3-small` | 1536 |
| Ollama | `nomic-embed-text` | 768 |
| Gemini | `gemini-embedding-exp-03-07` | 768 |
| NVIDIA NIM | `nvidia/llama-nemotron-embed-1b-v2` | 1024 |

**处理参数（可配置）：**
- `batch_size = 32`（每批处理 chunk 数）
- `tasks_limit = 3`（并发数，通过 asyncio.Semaphore 控制）
- `max_retries = 3`（指数退避重试）

关键代码：[provider.py#L340](AstrBot/astrbot/core/provider/provider.py#L340)

### 阶段 5：存储（Storage）

采用两层存储架构：

#### A. 向量存储 — FAISS

- 索引类型：`IndexFlatL2`（L2 欧氏距离）包裹 `IndexIDMap`
- 检索时 L2 距离归一化为 [0,1] 相似度分数
- 每个知识库一个 `index.faiss` 文件，持久化到磁盘

关键代码：[embedding_storage.py](AstrBot/astrbot/core/db/vec_db/faiss_impl/embedding_storage.py)

#### B. 文档/元数据存储 — SQLite

- 存储每个 chunk 的 UUID、文本内容、JSON 元数据（kb_id, kb_doc_id, chunk_index）
- 同时维护 **FTS5 全文搜索索引**（用于关键词检索）
- 每个知识库一个 `doc.db` 文件

关键代码：[document_storage.py](AstrBot/astrbot/core/db/vec_db/faiss_impl/document_storage.py)

#### C. 知识库元数据 — SQLite

- 全局 `kb.db` 文件，包含三张表：
  - `knowledge_bases`：知识库注册信息及配置
  - `kb_documents`：文档目录
  - `kb_media`：媒体目录

关键代码：[kb_db_sqlite.py](AstrBot/astrbot/core/knowledge_base/kb_db_sqlite.py)

## 三、检索管道（查询时）

查询时采用 **混合检索 + 多阶段排序**：

| 阶段 | 方法 | 默认 Top-K |
|------|------|-----------|
| 稠密检索 | FAISS 向量相似度（L2 距离） | 50 |
| 稀疏检索 | SQLite FTS5 / BM25 关键词匹配 | 50 |
| 融合排序 | RRF（Reciprocal Rank Fusion，k=60） | 20 |
| 重排序（可选） | Rerank 模型重打分 | 5（最终返回） |

### RRF 融合公式

```
score(d) = Σ (1 / (k + rank_i))  ，其中 k = 60
```

### 可选的重排序（Rerank）提供商

- vLLM Rerank
- Xinference Rerank
- 阿里百炼 Rerank
- NVIDIA Rerank

关键代码：
- 检索管理器：[manager.py](AstrBot/astrbot/core/knowledge_base/retrieval/manager.py)
- 稀疏检索：[sparse_retriever.py](AstrBot/astrbot/core/knowledge_base/retrieval/sparse_retriever.py)
- 分词器（jieba）：[tokenizer.py](AstrBot/astrbot/core/knowledge_base/retrieval/tokenizer.py)
- RRF 融合：[rank_fusion.py](AstrBot/astrbot/core/knowledge_base/retrieval/rank_fusion.py)

## 四、URL 内容清洗（可选）

URL 导入时可启用 LLM 清洗步骤：

1. 文本先经过 `RecursiveCharacterChunker` 分块
2. 每个 chunk 发送给 LLM（可配置 `cleaning_provider_id`）
3. LLM 根据 prompt 提取"信号"、丢弃"噪声"
4. 输出中识别 `<repaired_text>` 标签或 `<discard_chunk />` 信号
5. 并发处理，支持速率限制（默认 60 RPM）

关键代码：[kb_helper.py#L667](AstrBot/astrbot/core/knowledge_base/kb_helper.py#L667)、[prompts.py](AstrBot/astrbot/core/knowledge_base/prompts.py)

## 五、对话中的知识库集成

### 被动模式（默认，`kb_agentic_mode = False`）

每次 LLM 调用前自动查询知识库，将检索结果注入系统提示词 `[Related Knowledge Base Results]` 中。

### 主动/Agentic 模式（`kb_agentic_mode = True`）

不自动注入，而是为 LLM 注册 `astr_kb_search` 工具，由 LLM 通过 function calling 自行决定何时查询知识库。

关键代码：
- 主 Agent 集成：[astr_main_agent.py#L225](AstrBot/astrbot/core/astr_main_agent.py#L225)
- KB 工具：[knowledge_base_tools.py](AstrBot/astrbot/core/tools/knowledge_base_tools.py)

## 六、配置参数

### 全局配置（`config/default.py`）

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `kb_names` | `[]` | 默认查询的知识库名称列表 |
| `kb_fusion_top_k` | 20 | RRF 融合后的 top 结果数 |
| `kb_final_top_k` | 5 | 最终返回给 LLM 的结果数 |
| `kb_agentic_mode` | false | 是否使用工具式查询 |

### 每个知识库的独立配置（存储在 SQLite 中）

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `chunk_size` | 512 | 分块大小（字符） |
| `chunk_overlap` | 50 | 分块重叠（字符） |
| `top_k_dense` | 50 | 稠密检索返回数 |
| `top_k_sparse` | 50 | 稀疏检索返回数 |
| `top_m_final` | 5 | 最终返回数 |
| `embedding_provider_id` | 必填 | 嵌入提供商 |
| `rerank_provider_id` | 可选 | 重排序提供商 |

## 七、总结

```
上传文件 → 解析为文本 → 递归分块(512字/块)
    → 每块向量化（OpenAI/Ollama/Gemini/NVIDIA）
    → 向量存入 FAISS(L2) + 文本存入 SQLite(FTS5)

查询时 → 稠密检索(FAISS向量) + 稀疏检索(FTS5关键词)
    → RRF融合排序 → 可选Rerank → 返回Top-5 → 注入LLM上下文
```

**结论：上传的每个文件都会被完整解析 → 分块 → 逐块向量化 → 存入 FAISS 索引，是全量向量化，无选择性跳过。**
